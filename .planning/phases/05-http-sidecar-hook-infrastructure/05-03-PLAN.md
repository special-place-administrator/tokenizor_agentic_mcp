<task type="auto">
  <name>Task 1: Rewrite main.rs for CLI dispatch with sidecar spawn</name>
  <files>src/main.rs</files>
  <action>
Rewrite src/main.rs to add CLI dispatch and sidecar spawn. The key change: main() becomes a synchronous function that parses CLI args first, then routes to the appropriate handler. Only the MCP server path creates a tokio runtime.

Structure:
```rust
fn main() -> anyhow::Result<()> {
    let cli = Cli::parse();
    match cli.command {
        Some(Commands::Init) => cli::init::run_init(),
        Some(Commands::Hook { subcommand }) => cli::hook::run_hook(&subcommand),
        None => run_mcp_server(),
    }
}
```

The `run_mcp_server()` function contains the existing main() body (with `#[tokio::main]` or explicit runtime build), plus the sidecar spawn:

1. Initialize tracing (existing)
2. Auto-index logic (existing)
3. Spawn file watcher (existing)
4. **NEW: Spawn sidecar** — after watcher, before MCP serve:
   ```rust
   let bind_host = std::env::var("TOKENIZOR_SIDECAR_BIND")
       .unwrap_or_else(|_| "127.0.0.1".to_string());
   let sidecar_handle = sidecar::spawn_sidecar(Arc::clone(&index), &bind_host).await?;
   tracing::info!(port = sidecar_handle.port, "HTTP sidecar started");
   ```
5. Create MCP server and serve (existing)
6. **NEW: After MCP serve completes**, send shutdown signal to sidecar:
   ```rust
   let _ = sidecar_handle.shutdown_tx.send(());
   tracing::info!("sidecar shutdown signal sent");
   ```

CRITICAL design decisions:
- `fn main()` is NOT async (no `#[tokio::main]`). This avoids starting the tokio runtime for `tokenizor init` and `tokenizor hook` subcommands, per the research pitfall about unnecessary runtime overhead.
- `run_mcp_server()` creates its own tokio runtime explicitly via `tokio::runtime::Builder::new_multi_thread().enable_all().build()?.block_on(async { ... })`.
- Init and hook subcommands are synchronous — they MUST NOT call any async code or start tokio.
- Tracing is only initialized in the MCP server path. Hook subcommands MUST NOT call `observability::init_tracing()` (per HOOK-10 pitfall — tracing to stdout corrupts hook JSON output).
- Init command may optionally call `observability::init_tracing()` for diagnostic logging since its output goes to stderr, but it's simpler to skip it.

The sidecar spawns even when TOKENIZOR_AUTO_INDEX=false (the index is empty but the sidecar still serves empty results). This is deliberate — hooks should always have an endpoint to call.

<task type="auto">
  <name>Task 2: Integration tests for sidecar, hooks, and init</name>
  <files>tests/sidecar_integration.rs, tests/init_integration.rs</files>

1. Create tests/sidecar_integration.rs with the following integration tests. Each test creates a temporary directory, loads a LiveIndex with test files, spawns the sidecar, and verifies behavior:

   a. `test_sidecar_binds_ephemeral_port` (HOOK-01):
      - Create a LiveIndex with 1 Rust file
      - Call spawn_sidecar with "127.0.0.1"
      - Assert port > 0
      - Assert .tokenizor/sidecar.port file exists and contains the port
      - Assert .tokenizor/sidecar.pid file exists
      - Send shutdown and verify files are cleaned up (wait briefly for async cleanup)

   b. `test_health_endpoint_responds` (HOOK-01):
      - Spawn sidecar
      - Make sync HTTP GET to /health on the sidecar port
      - Parse response as JSON
      - Assert response contains "file_count" and "symbol_count" keys
      - Assert response latency < 50ms

   c. `test_outline_endpoint` (HOOK-01):
      - Index a Rust file with a known function `fn hello()`
      - Spawn sidecar
      - GET /outline?path=<relative_path>
      - Assert JSON response contains symbol with name "hello"

   d. `test_shared_index_mutation` (HOOK-02):
      - Spawn sidecar with a shared index
      - Verify /outline returns symbols for a file
      - Add a new file to the index via the same Arc (index.write().add_file(...))
      - Verify /outline now returns symbols for the new file too
      - This proves sidecar and code share the same Arc<LiveIndex>

   e. `test_hook_binary_latency` (HOOK-03):
      - Spawn sidecar
      - Write port file
      - Measure time to call run_hook(HookSubcommand::Read) with TOKENIZOR_HOOK_FILE_PATH set
      - Assert total time < 100ms (generous, actual should be <10ms)

   f. `test_hook_output_valid_json` (HOOK-10):
      - Spawn sidecar
      - For each subcommand (Read, Edit, Grep, SessionStart):
        - Capture stdout (redirect via test helper or call the JSON-building functions directly)
        - Parse output with serde_json
        - Assert "hookSpecificOutput" key exists
        - Assert "additionalContext" key exists

   g. `test_hook_failopen_valid_json` (HOOK-10):
      - Do NOT spawn sidecar (no port file)
      - Call run_hook for each subcommand
      - Verify output is valid JSON with empty additionalContext

   h. `test_repo_map_endpoint`:
      - Index 3 files
      - GET /repo-map
      - Assert response is JSON array with 3 entries, each with "path" and "symbol_count"

   Use `#[tokio::test(flavor = "multi_thread", worker_threads = 2)]` for all tests that spawn the sidecar.

   For HTTP calls in tests, use `std::net::TcpStream` with the same manual HTTP/1.1 GET pattern used in the hook binary (or use a small helper function shared between tests).

   For hook output capture, since run_hook writes to actual stdout, test the JSON-building helper functions directly (fail_open_json, success_json) rather than capturing process stdout. The process-level stdout test can be a single test that spawns the binary as a subprocess (similar to the existing test_stdout_purity pattern).

2. Create tests/init_integration.rs with integration tests:

   a. `test_init_writes_hooks`:
      - Set up a temp directory as a fake home dir
      - Override the settings path (pass as parameter or use env var)
      - Call the merge logic with a temp settings.json
      - Verify the output file contains PostToolUse and SessionStart hooks
      - Verify all 4 hook entries are present

   b. `test_init_idempotent`:
      - Run the merge logic twice on the same temp settings.json
      - Read file after first run, read after second run
      - Assert contents are byte-identical

   c. `test_init_preserves_other_hooks`:
      - Create a settings.json with an existing non-tokenizor PostToolUse hook
      - Run init merge
      - Verify the non-tokenizor hook is still present

   Note: init integration tests need the merge logic to accept a custom settings path for testing (not hardcoded to ~/.claude/settings.json). Expose a `pub fn merge_hooks_into_settings(settings_path: &Path, binary_path: &Path) -> anyhow::Result<()>` function that run_init wraps with the real paths.
