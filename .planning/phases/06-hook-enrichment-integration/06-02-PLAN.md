<objective>
Replace the Phase 5 env-var shim in the hook binary with stdin JSON parsing that reads Claude Code's native PostToolUse event payload, routes to the correct sidecar endpoint by tool_name, and converts absolute paths to relative. Update `tokenizor init` to write a single stdin-routed PostToolUse entry with auto-migration from old format.

Purpose: The hook binary is the bridge between Claude Code events and the sidecar. Stdin JSON parsing is the correct integration point (env vars were a Phase 5 temporary shim). The init command must register hooks that match the new routing approach.

Output: hook.rs with stdin parsing + routing, Write support added to CLI, init.rs with single-entry format + migration.
</objective>

    **In src/cli/mod.rs:**
    1. Add `Write` variant to `HookSubcommand` enum: `/// PostToolUse hook for the Write tool — confirms indexing of new file\nWrite,`
    2. Change `Hook { subcommand: HookSubcommand }` to `Hook { subcommand: Option<HookSubcommand> }` — makes subcommand optional so `tokenizor hook` (no subcommand) reads stdin for routing. The `#[command(subcommand)]` attribute on the field handles this with `Option<HookSubcommand>`.

       ```rust
       #[derive(serde::Deserialize, Default)]
       struct HookInput {
           tool_name: Option<String>,
           tool_input: Option<HookToolInput>,
           cwd: Option<String>,
       }

       #[derive(serde::Deserialize, Default)]
       struct HookToolInput {
           file_path: Option<String>,
           pattern: Option<String>,
           path: Option<String>,
       }
       ```
    5. Add `fn parse_stdin_input() -> HookInput`: read all lines from `std::io::stdin().lock().lines()`, concatenate, deserialize with `serde_json::from_str(&raw).unwrap_or_default()`.
    6. Add `pub fn relative_path(absolute: &str, cwd: &str) -> String`: use `std::path::Path::new(absolute).strip_prefix(cwd)`, map the result to forward-slash string via `.replace('\\', "/")`, fall back to `absolute.to_string()` on strip_prefix failure.
    7. Change `run_hook(subcommand: &HookSubcommand)` signature to `run_hook(subcommand: Option<&HookSubcommand>)`.
    8. At the top of `run_hook`: call `parse_stdin_input()`. If `subcommand` is `Some`, use it directly (backward compat for manual testing). If `subcommand` is `None`, determine the tool from `input.tool_name`.
    9. Update `endpoint_for` to accept the resolved tool info (either from subcommand or from stdin). Replace env var reads (`TOKENIZOR_HOOK_FILE_PATH`, `TOKENIZOR_HOOK_QUERY`) with stdin-derived values. For Read/Edit/Write: extract `input.tool_input.file_path`, convert to relative via `relative_path(file_path, cwd)`, URL-encode. For Grep: extract `input.tool_input.pattern`. For Write: route to `/impact` with `new_file=true` added to query.
    10. Update `event_name_for` to handle `HookSubcommand::Write` (returns "PostToolUse").
    11. Update existing unit tests for the new function signatures. Add new tests for `parse_stdin_input`, `relative_path`, and Write routing.
    12. Remove or gate the old `TOKENIZOR_HOOK_FILE_PATH`/`TOKENIZOR_HOOK_QUERY` env var reads — stdin is the primary source now. Old subcommands still work because the subcommand variant directly maps to the endpoint (no env vars needed when subcommand is explicit — but the file path still needs to come from somewhere for manual testing; for manual test mode, accept a `--path` flag on the subcommand or document that manual testing requires piping stdin JSON).

    **Key anti-patterns to avoid:**
    - Do NOT add a tokio runtime. All I/O stays synchronous (HOOK-10 constraint).
    - Do NOT print anything to stdout except the final JSON line.
    - Fail-open: if stdin is empty, unparseable, or missing fields, output empty additionalContext JSON gracefully.
