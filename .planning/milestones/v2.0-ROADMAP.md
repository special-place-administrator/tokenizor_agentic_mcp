# Roadmap: Tokenizor v2.0 — Deep Understanding

## Overview

Five phases in dependency order. Phases 1-2 deliver the highest-value agent-facing improvements (fewer round-trips, better navigation). Phases 3-4 add polish and visualization. Phase 5 adds temporal intelligence from git history.

The guiding principle: move from "where is this symbol?" to "what does this symbol need, who implements it, and what's risky about changing it?"

## Phases

- [x] **Phase 1: Recursive Type Resolution** — Transitive type closure in context bundles, depth-limited, cross-file (completed 2026-03-12)
- [ ] **Phase 2: Trait/Interface Implementation Mapping** — New xref kind, impl extraction across languages, find_implementations tool
- [ ] **Phase 3: Enriched File Context** — Import/export summaries from existing reverse index data
- [x] **Phase 4: Dependency Visualization** — Mermaid and DOT output formats for find_dependents (completed 2026-03-12)
- [ ] **Phase 5: Git Temporal Context** — Churn scores, co-change detection, risk metadata in file context

## Phase Details

### Phase 1: Recursive Type Resolution ✅
**Goal**: A single `get_context_bundle` call returns the focus symbol plus all custom type definitions needed to understand its signature — no more "what does UserConfig look like?" follow-up calls
**Depends on**: Nothing (builds on existing xref and symbol index)
**Requirements**: TYPR-01, TYPR-02, TYPR-03, TYPR-04, TYPR-05
**Status**: Completed 2026-03-12 (shipped in v0.6.0)
**Success Criteria**:
  1. ✅ `get_context_bundle` for a function using 3 custom types returns all 3 type definitions in a `dependencies` section
  2. ✅ Depth-2 recursion works: if `UserConfig` contains an `Address` field, `Address` definition is included
  3. ✅ Primitive/std types (String, Vec, Option, i32, etc.) are excluded from resolution
  4. ✅ Performance: bundle generation adds <50ms for typical functions
  5. ✅ All existing tests still pass; bundle output is backward-compatible (new section is additive)

**Implementation notes**: Resolution lives in `src/live_index/query.rs` (`type_refs_for_symbol`, `resolve_type_dependencies`). Formatting in `src/protocol/format.rs` (`format_type_dependencies`). Depth hardcoded at 2, max 15 dependencies. Types resolved: struct, enum, type, interface, class, trait.

**Key files**: `src/live_index/query.rs` (type resolution), `src/protocol/format.rs` (bundle formatter)

### Phase 2: Trait/Interface Implementation Mapping
**Goal**: Agents can ask "who implements this trait?" and "what traits does this struct implement?" — closing the loose-coupling navigation gap
**Depends on**: Phase 1 (type resolution infrastructure useful for impl discovery)
**Requirements**: IMPL-01, IMPL-02, IMPL-03, IMPL-04, IMPL-05
**Success Criteria**:
  1. `find_implementations("Display")` returns all `impl Display for X` blocks with file/line locations
  2. Reverse query: given a struct name, returns its trait implementations
  3. Works for Rust (impl Trait for Struct), TypeScript/Java (implements), Python (class inheritance), Go (interface satisfaction via method sets)
  4. Results respect OutputLimits
  5. New `ReferenceKind::Implements` is populated during xref extraction

**Key files**: `src/parsing/xref.rs` (extraction), `src/domain/index.rs` (ReferenceKind), `src/protocol/tools.rs` (new tool), `src/live_index/search.rs` (query method)

### Phase 3: Enriched File Context
**Goal**: `get_file_context` tells the agent "this file depends on 3 symbols from combat.rs and provides 2 public functions used by main.rs" — immediate interface understanding
**Depends on**: Nothing (uses existing reverse index)
**Requirements**: ECTX-01, ECTX-02, ECTX-03, ECTX-04
**Success Criteria**:
  1. Enriched view includes grouped import sources with counts
  2. Enriched view includes grouped export consumers with counts
  3. No new indexing — data sourced entirely from existing reverse index and xref tables
  4. Output stays within existing token budget constraints

**Key files**: `src/protocol/format.rs` (file context formatter), `src/live_index/search.rs` (data queries)

### Phase 4: Dependency Visualization ✅
**Goal**: `find_dependents` can output Mermaid or DOT graphs for visual architecture understanding
**Depends on**: Nothing (formatter-only change)
**Requirements**: DVIZ-01, DVIZ-02, DVIZ-03, DVIZ-04
**Status**: Completed 2026-03-12 (shipped in v0.5.0)
**Success Criteria**:
  1. ✅ `find_dependents` with `format: "mermaid"` produces valid Mermaid flowchart syntax
  2. ✅ `find_dependents` with `format: "dot"` produces valid Graphviz digraph syntax
  3. ✅ Default format remains `text` — fully backward compatible
  4. ✅ Graphs respect limit/max_per_file bounds

**Key files**: `src/protocol/format.rs` (formatter paths), `src/protocol/tools.rs` (format parameter on input)

### Phase 5: Git Temporal Context
**Goal**: File context includes churn frequency, last modification date, and co-change partners — agents know which files are volatile and which are stable
**Depends on**: Phase 3 (enriched file context is the output surface)
**Requirements**: GTMP-01, GTMP-02, GTMP-03, GTMP-04, GTMP-05
**Success Criteria**:
  1. Git history parsed on startup in <500ms for repos with <1000 commits
  2. `get_file_context` includes `last_modified`, `churn_frequency`, `often_changed_with` fields
  3. Non-git repos degrade gracefully — fields simply absent
  4. Co-change threshold configurable (default: files appearing together in >30% of commits)

**Key files**: New `src/git/` module, `src/live_index/store.rs` (metadata storage), `src/protocol/format.rs` (output)

## Explicitly Out of Scope

These were considered and deliberately excluded:

| Feature | Reason |
|---------|--------|
| Embedding/semantic search | +80MB binary, breaks zero-dependency local-first principle |
| Virtual diff / shadow workspace | Requires per-language type checking — building half a compiler |
| Cross-language polyglot linking | Fragile URI heuristics, high maintenance, niche use case |
| Cyclomatic complexity scoring | Tangential to navigation mission, covered by clippy/SonarQube |
| Architectural policy enforcement | Custom linter runtime, separate tool concern |
