<!-- Port file interface from Plan 01 (convention, not import — hook uses its own file read) -->

Port file convention:
- Location: `.tokenizor/sidecar.port` (relative to working directory)
- Content: ASCII decimal port number, no trailing newline
- If file missing or unparseable: sidecar is not running

Sidecar endpoint conventions (from CONTEXT.md locked decisions):
- GET /health — no params, returns JSON health object
- GET /outline?path=<relative_path> — returns JSON array of symbols
- GET /impact?path=<relative_path> — returns JSON array of dependents
- GET /symbol-context?name=<name> — returns JSON array of references
- GET /repo-map — no params, returns JSON array of files with symbol counts

Claude Code hooks schema (from RESEARCH.md, official docs):
```json
{
  "hooks": {
    "PostToolUse": [
      {
        "matcher": "Read",
        "hooks": [{ "type": "command", "command": "/path/to/tokenizor hook read", "timeout": 5 }]
      }
    ]
  }
}
```

Hook stdout output format:
```json
{
  "hookSpecificOutput": {
    "hookEventName": "PostToolUse",
    "additionalContext": "..."
  }
}
```

2. Create src/cli/mod.rs:
   - `pub mod hook; pub mod init;`
   - Define clap types:
   ```rust
   use clap::{Parser, Subcommand};

   #[derive(Parser)]
   #[command(name = "tokenizor", about = "Tokenizor MCP server and hook system")]
   pub struct Cli {
       #[command(subcommand)]
       pub command: Option<Commands>,
   }

   #[derive(Subcommand)]
   pub enum Commands {
       /// Install hooks into ~/.claude/settings.json
       Init,
       /// Hook subcommands called by Claude Code
       Hook {
           #[command(subcommand)]
           subcommand: HookSubcommand,
       },
   }

   #[derive(Subcommand, Debug, Clone)]
   pub enum HookSubcommand {
       /// PostToolUse hook for Read
       Read,
       /// PostToolUse hook for Edit/Write
       Edit,
       /// PostToolUse hook for Grep
       Grep,
       /// SessionStart hook for repo map
       #[command(name = "session-start")]
       SessionStart,
   }
   ```

3. Create src/cli/hook.rs:
   - `pub fn run_hook(subcommand: &HookSubcommand) -> anyhow::Result<()>`
   - This function MUST NOT initialize tracing (no tracing to stdout). It MUST NOT create a tokio runtime.
   - Step 1: Read `.tokenizor/sidecar.port` file. If missing or unparseable, output fail-open JSON and return Ok.
   - Step 2: Determine endpoint path based on subcommand:
     - Read -> GET /outline (reads TOKENIZOR_HOOK_FILE_PATH env var set by Claude Code via stdin/env, or falls back to empty)
     - Edit -> GET /impact (same env var)
     - Grep -> GET /symbol-context (reads TOKENIZOR_HOOK_QUERY env var, or empty)
     - SessionStart -> GET /repo-map
     Note: In Phase 5, the actual input parsing from Claude Code's hook invocation (stdin JSON with tool_input) is deferred to Phase 6. For now, the hook binary reads a simple env var for the file path. The important thing is the HTTP plumbing works end-to-end.
   - Step 3: Build URL `http://127.0.0.1:{port}{endpoint_path}` with query params
   - Step 4: Make sync HTTP GET using `std::net::TcpStream::connect_timeout` with 50ms timeout:
     - Manually construct HTTP/1.1 GET request: `GET {path}?{query} HTTP/1.1\r\nHost: 127.0.0.1:{port}\r\nConnection: close\r\n\r\n`
     - Read response, parse body after `\r\n\r\n` separator
     - On any error (connect refused, timeout, parse error), fall through to fail-open
   - Step 5: Format output JSON:
     - Success: `{"hookSpecificOutput":{"hookEventName":"PostToolUse","additionalContext":"<response_body>"}}`
     - Fail-open: `{"hookSpecificOutput":{"hookEventName":"PostToolUse","additionalContext":""}}`
     - For SessionStart subcommand, hookEventName should be "SessionStart"
   - Step 6: Print JSON to stdout via `println!` and return Ok(())

   Helper: `fn fail_open_json(event_name: &str) -> String` — returns the empty additionalContext JSON string.
   Helper: `fn success_json(event_name: &str, context: &str) -> String` — returns JSON with the context.

   CRITICAL (HOOK-10): The ONLY thing written to stdout is the final JSON line. No tracing, no debug prints, no eprintln that accidentally goes to stdout. Use `eprintln!` for any debug/error messages (but prefer silent operation).

   Unit tests (these test the JSON output format, not the HTTP call):
   - `test_fail_open_json_is_valid` — parse fail_open_json output with serde_json, verify structure
   - `test_success_json_is_valid` — parse success_json output, verify additionalContext contains expected string
   - `test_hook_subcommand_to_endpoint` — verify each subcommand maps to the correct endpoint path
  </action>

<task type="auto">
  <name>Task 2: tokenizor init command — idempotent settings.json merge</name>
  <files>src/cli/init.rs</files>
  <action>
1. Create src/cli/init.rs:
   - `pub fn run_init() -> anyhow::Result<()>`
   - Step 1: Discover binary path via `std::env::current_exe()`. If path contains "node_modules" or ends in ".cmd", log a warning to stderr but continue.
   - Step 2: Resolve settings path: `dirs::home_dir().unwrap().join(".claude").join("settings.json")`. Create `.claude/` directory if it doesn't exist.
   - Step 3: Read existing settings.json or create empty `{}` if file doesn't exist.
   - Step 4: Parse as `serde_json::Value`.
   - Step 5: Ensure `hooks` key exists as an object. If missing, create it.
   - Step 6: Build tokenizor hook entries. The identification marker is the substring "tokenizor hook" in the command string. For each hook event type:
     - PostToolUse: 3 entries (Read, Edit|Write, Grep) — each with matcher regex and command path
     - SessionStart: 1 entry (session-start)
   - Step 7: Merge strategy for each event type key (e.g., "PostToolUse"):
     - If key doesn't exist, create it as an array with the tokenizor entry
     - If key exists as an array, filter OUT any existing entries whose `hooks[].command` contains "tokenizor hook", then append the fresh tokenizor entries
     - This ensures idempotency: old tokenizor entries are replaced, non-tokenizor entries are preserved
   - Step 8: Write settings.json back with `serde_json::to_string_pretty` (4-space indent)
   - Step 9: Create `.tokenizor/` directory in current working directory if it doesn't exist (runtime will need it for port/PID files)
   - Step 10: Print to stderr: "tokenizor hooks installed in {settings_path}" (NOT stdout — stdout must remain clean)

   The exact hook entries to write (per RESEARCH.md authoritative format):
   ```json
   {
     "PostToolUse": [
       {
         "matcher": "Read",
         "hooks": [{"type": "command", "command": "{binary_path} hook read", "timeout": 5}]
       },
       {
         "matcher": "Edit|Write",
         "hooks": [{"type": "command", "command": "{binary_path} hook edit", "timeout": 5}]
       },
       {
         "matcher": "Grep",
         "hooks": [{"type": "command", "command": "{binary_path} hook grep", "timeout": 5}]
       }
     ],
     "SessionStart": [
       {
         "matcher": "startup|resume",
         "hooks": [{"type": "command", "command": "{binary_path} hook session-start", "timeout": 5}]
       }
     ]
   }
   ```

   On Windows, the binary_path must use forward slashes or escaped backslashes in the JSON command string. Use `binary_path.display().to_string().replace('\\', "/")` for cross-platform compatibility.

   Unit tests:
   - `test_init_creates_hooks_in_empty_settings` — start with `{}`, run merge logic, verify hooks section has 4 entries (3 PostToolUse + 1 SessionStart)
   - `test_init_preserves_existing_hooks` — start with settings containing a non-tokenizor PostToolUse hook, run merge, verify the existing hook is still present alongside tokenizor hooks
   - `test_init_idempotent` — run merge twice on the same Value, verify output is identical both times
   - `test_init_replaces_stale_tokenizor_entries` — start with settings containing an old tokenizor hook entry (different binary path), run merge, verify old entry is gone and new entry is present
  </action>
