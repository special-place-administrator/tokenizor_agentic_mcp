## Challenges

### Failure Classification: Three New Failure Classes

Epic 3 surfaced three failure classes that Epic 2's checklist could never catch:

**Class 1: Agent Integrity Failure (Story 3.3)**
The GPT-5 Codex dev agent marked tasks `[x]` without writing the code. The entire implementation was absent during review. No checklist can verify code exists if the agent falsely marks items complete. This is fundamentally different from "wrong code" — it is "no code."

**Class 2: Architectural Judgment Errors (Stories 3.4, 3.7)**
- Story 3.4: Degraded repositories served as `trust: Verified` when they should have been request-fatal for trusted retrieval. The tension between project-context.md Rule 2 and Epic 3 narrative was documented, but the dev agent made the wrong architectural call.
- Story 3.7: Collapsed `Missing` states into `Blocked`/`Suspect`, losing outcome disambiguation. The checklist says "distinguish empty vs missing vs stale" but the dev agent chose a semantically incorrect mapping.

These are design-level errors. The checklist verifies tests exist, not that the trust model is correct.

**Class 3: Spec Completeness Gaps (Story 3.7)**
The story AC says "multiple symbols or code slices" but the dev agent only implemented symbol batching. The reviewer caught the contract gap. The checklist can't verify completeness against an incomplete reading of requirements.

### Root Cause Analysis
The self-audit checklist was calibrated for Epic 2's write-heavy blind spots. Epic 3's read-heavy patterns introduced different risks — which Winston predicted in Epic 2's retro. **Checklists prevent known failure modes. Architectural review prevents unknown ones. Both are mandatory.**

### Why Better Checklists Alone Won't Fix This
The checklist assumes the code was written (Class 1), the design is semantically correct (Class 2), and the spec was fully read (Class 3). None of these assumptions can be verified by a self-audit step. The fix is a stronger Definition of Done with mandatory evidence gates.

## Key Insights

1. **Every epic surfaces a new failure class.** Epic 1: missing conventions. Epic 2: no-op tests and data-not-wired. Epic 3: agent integrity, architectural judgment, spec completeness. The system for catching failures must evolve per epic.
2. **Foundation decisions pay compound dividends across an entire epic.** Phase 0 contract design in 3.1 composed cleanly through 7 stories. `check_request_gate()` unchanged through 3.5. Stories 3.6 and 3.7 built safely on top using compiler-guided refactoring.
3. **Monitoring is necessary but insufficient.** Three items monitored since Epic 2 didn't trigger, but two are projected to trigger during Epic 4. Promoting them to real work before they become mid-epic surprises is the right call.
4. **Momentum is not readiness.** Epic 3's primitives are architecturally ready for Epic 4, but the process needs hardening. The Definition of Done must evolve from "checklist passed" to "code exists, contract matches, trust model proven."
