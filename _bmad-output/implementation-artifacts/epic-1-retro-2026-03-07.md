## Challenges

- **Story 1.3 needed significant hardening** — initial implementation lacked atomic writes, lock-based concurrency, and strict Git root detection. Caught in review.
- **Same-model dev+review blind spots** — GPT-5 Codex wrote and reviewed 5/7 stories. Cross-model review on Story 1.7 (Claude Opus 4.6) caught the most issues (3 MEDIUM + 3 LOW).
- **Recurring structural issues across reviews:**
  - Production `expect()` calls needing proper error propagation (1.7)
  - 402-line function needing decomposition (1.7)
  - Undocumented `unsafe` FFI block (1.7)
  - Missing durability primitives on first implementation pass (1.3)
- **Missing BMAD validator task** — noted in every single story; all 7 manually validated
- **Missing project-context.md** — conventions rediscovered per-story instead of inherited
- **SpacetimeDB persistence entirely deferred** — all control-plane writes return pending_write_error()

## Key Insights

1. **Cross-model reviews catch category problems that same-model reviews share as blind spots.** Different models bring different priors about what's "acceptable" regarding safety, decomposition, and documentation. The empirical evidence from our own stories is clear.
2. **Build business logic first, wire infrastructure later.** The `ControlPlane` trait boundary is architectural insurance. Building indexing against local persistence first means proving business logic works before introducing infrastructure complexity.
3. **Small tooling gaps compound across every story in an epic.** Missing validator and project-context.md were each noted 7 times — once per story. Fix once, benefit everywhere.
4. **Epic 1 velocity is not the baseline for Epic 2.** Epic 1 was structural plumbing — config, CLI wiring, registry CRUD, read-only queries. Epic 2 is algorithmic — indexing, parsing, content-addressable storage, idempotency. Different work, different pace.

## Process Improvements (Action Items)

### 1. Adopt Tiered Cross-Model Code Reviews

**Owner:** Bob (Scrum Master)
**Success criteria:** Enforced during story creation

| Risk Tier | Criteria | Review Model |
|-----------|----------|-------------|
| High | State mutations, concurrency, FFI, persistence, indexing engine | Cross-model mandatory |
| Low | Read-only queries, CLI wiring, status reporting | Same-model acceptable |

### 2. Add Structural Review Checklist

**Owner:** Bob (Scrum Master)
**Success criteria:** Embedded in story review workflow

Checklist items (derived from Epic 1 findings):
- [ ] Function length limits — no functions over ~100 lines without justification
- [ ] Panic-free paths — no production `expect()` or `unwrap()` without `// SAFETY:` justification
- [ ] Unsafe documentation — all `unsafe` blocks have `// SAFETY:` comments
- [ ] Durability primitives — state mutations use atomic writes and appropriate locking
